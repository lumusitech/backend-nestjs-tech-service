# Revisión del Proyecto — Tech Service Backend

## Características del Proyecto

### Stack tecnológico
- **NestJS 11** + **TypeScript 6** + **TypeORM** + **PostgreSQL 16**
- **Auth:** JWT + Passport (strategy pattern)
- **WebSockets:** Socket.IO para notificaciones real-time
- **Pagos:** MercadoPago SDK (strategy pattern)
- **PDFs:** pdfkit (billing + reports)
- **Validación:** class-validator + class-transformer
- **Documentación:** Swagger/OpenAPI integrado

### Arquitectura modular (16 módulos)
auth, users, clients, suppliers, service-types, work-orders, payments, finances, notifications, portal, reports, billing, pending-items, inquiries, user-preferences, common

### Módulos notables
- **Work Orders** — módulo core con 4 entidades (work-order, task, material, note), tracking code, filtros avanzados
- **Notifications** — WebSocket gateway + EventEmitter2 (listener con 13 eventos tipados)
- **Billing** — Strategy pattern con interfaz `BillingProvider` + stub ARCA/AFIP + PDF de facturas
- **Payments** — Strategy pattern con 3 providers (cash, transfer, mercadopago) e interfaz `PaymentProvider`
- **Portal** — Endpoint público sin auth para tracking por código
- **Reports** — BFF pattern con PDFs generados en backend

## Buenas Prácticas Observadas

### Arquitectura
- **BaseEntity** con UUID, timestamps y soft delete (`@DeleteDateColumn`)
- **Separación por módulos** completa: controller / service / entity / dto / enums
- **DTOs separados** para create, update, filter (PartialType para updates)
- **PaginationDto genérico** con `PaginatedResponseDto<T>` reutilizable
- **Common module** como módulo global con helpers compartidos

### Seguridad
- **JwtAuthGuard global** con decorador `@Public()` para rutas abiertas
- **RolesGuard** con decorador `@Roles(UserRole.ADMIN)` por controlador
- **ValidationPipe global** con `whitelist`, `forbidNonWhitelisted`, `transform`
- **Soft delete** en lugar de hard delete por defecto
- **Secrets por env** — `.env.template` documentado, `.env` en `.gitignore`
- **CORS configurado** solo para `localhost:4200`
- **Dependabot** configurado con agrupación por ecosistema

### Código
- **Tipado estricto** — `strict: true` en tsconfig
- **ESLint flat config** con type-checked rules + reglas relajadas para tests
- **Prettier** integrado en ESLint
- **Naming consistente** — kebab-case para archivos, PascalCase para clases
- **Decoradores custom** — `@CurrentUser()`, `@Public()`, `@Roles()`
- **Exception filter global** con formato de respuesta uniforme
- **TransformInterceptor** — envuelve responses en `{ statusCode, data, timestamp }`
- **Swagger** documentado en controllers Y entities (`@ApiProperty`)

### Testing
- **Unit tests** por servicio (*.spec.ts) en cada módulo
- **E2E tests** por módulo (14 archivos de e2e)
- **Acceptance tests** — flujos completos (billing-flow, work-order-flow)
- **Test helpers** — `mock-query-builder.helper.ts` para tests de TypeORM
- **Scripts dedicados** — `test:unit`, `test:e2e`, `test:acceptance`, `test:all`

### Infraestructura
- **Docker Compose** con PostgreSQL + pgAdmin + healthcheck
- **Migrations** — 13 migraciones TypeORM (no `synchronize: true`)
- **Seeds** — 13 archivos de seeds para datos de prueba
- **Data source** separado para CLI de TypeORM

### Event-Driven
- **EventEmitter2** para desacoplamiento entre módulos
- **Listener tipado** con interfaces para cada evento (`WorkOrderCreatedEvent`, etc.)
- **WebSocket gateway** con autenticación JWT por token

## Posibles Mejoras a Futuro

### Seguridad
- **CORS** — `origin: '*'` en WebSocket gateway es abierto; restringir a orígenes conocidos
- **Rate limiting** — No hay `@nestjs/throttler` configurado; vulnerable a brute force en auth
- **Helmet** — No se usa `helmet` para headers de seguridad HTTP
- **Refresh tokens** — Solo access token JWT, sin refresh token para sesiones largas
- **Password policy** — No hay validación de fuerza de contraseña en creación de usuarios

### Arquitectura
- **Logger** — Uso de `console.log` en algunos places; considerar `@nestjs/common` Logger o un logger estructurado (pino/winston)
- **PaginationDto sortBy** — No valida contra columnas permitidas; podría permitir inyección de ordenamiento
- **Global prefix** — `api` hardcoded en `main.ts`; pasar a variable de entorno
- **Config validation** — No hay `validationSchema` (Joi/zod) en `ConfigModule.forRoot()`; variables faltantes fallan silenciosamente en runtime

### Testing
- **Cobertura** — No hay threshold mínimo configurado en Jest (`coverageThreshold`)
- **E2E setup** — No hay `globalSetup` / `globalTeardown` visible para preparar DB de tests
- **Tests de controllers** — Solo tests de services; controllers y guards no tienen unit tests
- **Tests de interceptors/filters** — `TransformInterceptor` y `HttpExceptionFilter` sin tests

### Operaciones
- **Health check endpoint** — No hay `/health` o `/readiness` para monitoreo
- **Graceful shutdown** — No hay manejo de `SIGTERM`/`SIGINT` para cerrar conexiones limpiamente
- **Structured logging** — Agregar requestId, userId, etc. al logger para trazabilidad
- **APM/Metrics** — No hay integración con Prometheus, Datadog, etc.

### Code Quality
- **ESLint `no-explicit-any: off`** — Contradice la regla del AGENTS.md de no usar `any`
- **Object.assign en updates** — `clients.service.ts:76` usa `Object.assign`; considerar repositorio `preload` de TypeORM
- **Notifications listener** — Archivo de 379 líneas; podría dividirse por dominio (work-orders, payments, inquiries)
- **Migrations acumuladas** — Hay 2 migraciones "Init" al final; considerar squashing antes de producción

### Funcionalidad pendiente (según AGENTS.md)
- **ARCA/AFIP real** — Stub listo, falta conectar WSFEv1 con certificados
- **MercadoPago webhook** — Falta endpoint con verificación de firma
- **Scheduled tasks** — `@nestjs/schedule` registrado pero no se ven cron jobs implementados
